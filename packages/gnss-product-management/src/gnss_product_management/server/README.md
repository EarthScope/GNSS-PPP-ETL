# `server/`: The Communication Specialists

Right, this `server/` directory is where we keep our communications experts – the code responsible for talking to all sorts of remote (and even local) data repositories. Whether it's an old-school FTP server or a modern HTTP web host, the modules here provide the means to list directories and download files in a consistent, reliable manner.

The core idea here is to abstract away the messy details of different network protocols behind a clean, unified interface.

### `protocol.py`: The Universal Translator (`DirectoryAdapter` Protocol)

This file defines the `DirectoryAdapter` Protocol, which is essentially a contract. Any class that wants to be a "server adapter" must implement the methods defined here:

*   `can_connect()`: To check if a connection can be established.
*   `list_directory()`: To get a list of file and directory names from a specified path.
*   `download_file()`: To fetch a specific file and save it locally.

This Protocol is absolutely critical because it allows the higher-level components, like our `ResourceFetcher`, to interact with *any* type of server without needing to know the specific quirks of FTP or HTTP. It's a grand bit of design, ensuring flexibility and extensibility.

### `ftp.py`: The Old Salt (`FTPAdapter`)

This module contains the implementation for communicating with FTP (File Transfer Protocol) and FTPS (FTP Secure) servers.

*   It handles the complexities of establishing and authenticating connections, including trying plain FTP and falling back to FTPS if necessary (or strictly enforcing FTPS).
*   It provides the concrete logic for listing directories and robustly downloading files using the `ftplib` standard library.
*   The `FTPAdapter` class wraps these functionalities, making it compliant with the `DirectoryAdapter` Protocol.

### `http.py`: The Modern Man (`HTTPAdapter`)

This module provides the implementation for interacting with HTTP and HTTPS web servers. It uses the excellent `fsspec` library, which offers a uniform Pythonic interface to various filesystems.

*   It's particularly clever in how it handles directory listings: for many HTTP servers, it parses the HTML content of a directory listing page (often generated by Apache or Nginx) to extract filenames from `<a>` tags. This is a brilliant workaround for HTTP not having a native "list directory" command.
*   It handles caching of HTTP directory listings to improve performance for repeated requests.
*   The `HTTPAdapter` class implements the `DirectoryAdapter` Protocol, allowing it to seamlessly integrate into our fetching mechanism.

### `local.py`: The Homebody (`LocalAdapter` - implied)

Though not explicitly detailed, the existence of `local.py` strongly implies an adapter for local filesystem access. This would implement the `DirectoryAdapter` Protocol by treating the local disk as a "server," allowing local file operations (listing, reading) to be handled through the same generalized interface. This unified approach simplifies the `ResourceFetcher` as it doesn't need to distinguish between local and remote sources at a fundamental level.

In summary, the `server/` module is the backbone of our data acquisition. These adapters, guided by the `DirectoryAdapter` Protocol, are the communication specialists that allow our system to speak the language of various data hosts, ensuring we can always find and retrieve the GNSS products we need, regardless of where they're tucked away. They're essential cogs in the ETL machine, keeping the data flowing!
